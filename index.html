新学期班会

发表于 2020-03-05

学习小节

  度过了一个学期之后，对大学的概念愈加清晰。大学说是很空的吧，他也有很多的课程需要学习；说大学任务很紧，比起高中来说，课程表空很多了，有很多时间是由我们自己来决定使用的。想想自己在寝室里的日子，能不出门就不出门，出门也就上课吃饭。在寝室里，大部分时间都是用来与计算机玩耍，正事也是有做，但游戏也玩的不少，时间的分配上还是要更加合理一点。玩归玩，至少课程不能落下。不能只沉浸在大学的自由里面，毕竟大学还是学习的地方，挂科还是很难受的，平时一定要时间花上去。学习是首位，规划好学习与放松，认真听讲，这样就不用连夜复习，在考试周之前疯狂复习了。
  大学里我们也不仅仅学习专业知识(虽然计算机的部分这学期还没开始),我们现在不在家里面，不是躺在床上，醒来就有饭吃；洗完澡把衣服一丢，第二天洗好的衣服就会挂在阳台上。大学里我们也要学会照顾自己，学会掌握生活的技能，有空打扫干净自己的位置，勤丢垃圾，学会早上自觉地收拾好床铺，不要只是把衣服丢在角落里，等着洗衣机空出来，偶尔也要自己洗衣洗衣服，拖拖地，把寝室的氛围搞好。发展一下自己的兴趣，未未来做做准备。

  所以在家还是很舒服的，抓紧享受，回去很多事就要自己做了 =-=

寒假见闻

  寒假怎么说呢，还是很倒霉的。回到家的第二天就又出发前往贵州，先是等了一天的长途客车，结果耳机莫名其妙的裂开了。(之后它的命运也很坎坷，被我翻身压倒了，彻底裂开。)客车开了很久，窗上都是水，车内环境好差，空气又不好，还挤。路上一半身份证还不见了，糟糕死了。到了目的地还发现那不是目的地…骗我们到目的地附近的地方，就退我们那么点钱叫我们搭车，时间还刚好在客运站最后一班车出去之后。真是太巧了，太棒了(*****).然后就开始了乡下隔离，没有厕所，没有信号，房间通风很不错，凉嗖嗖的…
  接下来就是拿着手机混日子，总算是可以回去了。坐的火车，票是到的那天买的，那时候还差点买不到，结果当天一节车里面都没几个人。
  到家之后就一直宅家里坐自己的事情，敲敲代码，打打游戏，看看动漫。

线上学习情况&新学期计划

  • 线上学习
      学习情况就是跟着学 =-= 这部分不知道能讲啥。基本进度跟着教师，不过有些作业还么有写。虽然有时候会卡，但基本都还好。
  • 新学期计划
      计划这种东西还是很难搞得，学习这部分来说，教学进度不是我们决定的事情，能做到的就是跟上，巩固。其他的话,计划在这个学期能够多学一些技术性的东西，除了学校教的C语言课程之外，学一些关于计算机的事情，多参加一下这方面的活动之类的(不能只是一个宅男)

HTTPS

HTTP缺陷:

  1. 明文通信，可能被窃听
  2. 身份易被伪装
  3. 报文有风险被篡改

HTTPS 并不是新协议，而是让 HTTP 先和 SSL（Secure Sockets Layer）通信，再由 SSL 和 TCP 通信，也就是说 HTTPS 使用了隧道进行通信。

服务器认证阶段:

  1. 客户端向服务器发送一个开始信息“Hello”以便开始一个新的会话连接
  2. 服务器根据客户的信息确定是否需要生成新的主密钥，如需要则服务器在响应客户的“Hello”信息时将包含生成主密钥所需的信息
  3. 客户根据收到的服务器响应信息，产生一个主密钥，并用服务器的公开密钥加密后传给服务器
  4. 服务器回复该主密钥，并返回给客户一个用主密钥认证的信息，以此让客户认证服务器。
用户认证

用户认证阶段:
在此之前，服务器已经通过了客户认证，这一阶段主要完成对客户的认证。经认证的服务器发送一个提问给客户，客户则返回（数字）签名后的提问和其公开密钥，从而向服务器提供认证。

加密

对称加密: 使用同一密钥(优:运算快;缺点:无法将密钥安全传输)
不对称加密:使用不同密钥(优:安全传输密钥;缺:运算速度慢)

HTTPS 不对称加密密钥，再用对称密钥通信

认证

通过使用 证书 来对通信方进行认证。

数字证书认证机构（CA，Certificate Authority）是客户端与服务器双方都可信赖的第三方机构。

服务器的运营人员向 CA 提出公开密钥的申请，CA 在判明提出申请者的身份之后，会对已申请的公开密钥做数字签名，然后分配这个已签名的公开密钥，并将该公开密钥放入公开密钥证书后绑定在一起。

进行 HTTPS 通信时，服务器会把证书发送给客户端。客户端取得其中的公开密钥之后，先使用数字签名进行验证，如果验证通过，就可以开始通信了。

HTTPS缺点

  1. 因为需要进行加密解密等过程，因此速度会更慢；
  2. 需要支付证书授权的高额费用。

定义于 Uniform Resource Identifier (URI): Generic Syntax
URI: 统一资源(标识)符 包括URL和URN
URL: 统一资源(定位)符 提供路径 用具体路径来定位资源,大部分网址都是URL
URN: 统一资源名称 代表资源名称 在某一命名空间里通过表识资源名称来定位该资源
运用实例:磁力链接(baidu)

磁力链接:
‘磁力链接不基于文档的IP地址或定位符，而是在分布式数据库中，通过散列函数值来识别、搜索来下载文档。因为不依赖一个处于启动状态的主机来下载文档，所以特别适用没有中心服务器的对等网络。磁力链接由一组参数组成，参数间的顺序没有讲究，其格式与在HTTP链接末尾的查询字符串相同。通常是一个特定文件的内容散列函数值形成的URN’
通过文件内的代码生成一个独一无二的ID来指定文件，与地址无关。

发表于 2019-11-22 更新于 2019-12-01
  1. var a=function(){}是一个隐式函数定义，则a会有一个原型对象。a.b=1在a中存放了一个值为1的属性。a.prototype.X=Y可以向原型中添加属性。a.b访问了a中的属性，所以结果是1。而new a().b用new创建了新对象与原来的a无关，所以new a().b的b!=a.b=1的b。新对象中无b，向它的原型中搜索该值，结果为2。下面同上。
  1. var bar = new Foo()调用了Foo,返回了一个对象{a:2},该对象a属性的值为2,所以输出为2
  1. Object.create(null);创建了函数的原型(null),null中无"tostring",返回false。而{a:1}有原型，在查询"tostring" in map时，顺着原型链查到了"tostring"，结果为true,"a" in map直接被查询到，返回true。
  1. Object对象的prototype原型的toString方法,将this指向obj并调用。取输出结果的第八位至倒数第一位。即obj的数据类型
